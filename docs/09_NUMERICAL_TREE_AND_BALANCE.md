# 数值树与平衡验收

## 1. 权威数据源

数值调整按以下顺序取值，后者不能覆盖前者：

1. `data/config/numerical_tree.json`：效果点、预算区间和模拟硬门槛。
2. `data/cards`、`data/enemies`、`data/relics`、`data/config`：游戏实际数值。
3. `scripts/tools/NumericalTreeAuditor.gd`：静态预算审计。
4. `scripts/tools/BalanceSimulator.gd`：真实战斗核心驱动的完整跑团模拟。
5. 本文：公式、调参流程和当前基线的可读说明。

任何数值改动都必须同时修改实际数据和设计注释。报告中的 `warning` 是硬越界；`advisory` 是仍在容差内、但后续应观察的轻微偏差。

本文记录当前可复现的数值与路线验收基线，不代表全部产品需求、内容量或 UI 已完成。未完成的 REQ、正式 UI/演出和发布工作仍按交付清单推进。

## 2. 卡牌效果点

效果点用于比较不同效果，不直接显示给玩家。当前核心换算如下：

| 效果 | 效果点 | 说明 |
| --- | ---: | --- |
| 1 直接伤害 | 1.00 | 作为基准单位。 |
| 1 护甲 | 0.88 | 护甲回合结束清空，因此略低于伤害。 |
| 抽 1 张牌 | 4.60 | 包含卡位质量和组合启动价值。 |
| 获得 1 能量 | 7.20 | 约等于额外打出一张中等 1 费牌。 |
| 获得 1 势能 | 2.70 | 是延迟资源，不等同能量。 |
| 失去 1 势能 | -2.40 | 比获得值略低，避免资源转换牌被低估。 |
| 失去 1 生命 | -1.70 | 连续战斗中会累积，因此作为真实代价。 |
| 弃牌堆加入 1 张灼伤 | -3.20 | 延迟污染。加入手牌为 -5.00。 |
| 抽牌后消耗 | 总分 x 0.78 | 一次性爆发不能按可循环效果计满。 |
| 能力牌长期效果 | 总分 x 1.32 | 折算后续回合收益，同时保留准备回合成本。 |
| 条件效果 | 条件部分 x 0.72 | 以并非每次都满足为前提。 |

目标倍率：随机敌人为 `0.82`，全体敌人为 `1.85`。升级降费 1 点按 `4.80` 效果点计算。带势能成长的伤害以 2 点预期势能估值；`bonus_if_momentum_at_least` 只对条件奖励乘 `0.72`，基础伤害始终按满值计入，自动战斗 AI 也使用相同门槛。

## 5. 敌人与遭遇

敌人预算按整场遭遇审计。单个双敌战允许总峰值高于单怪，但不能把两个最大行动视作每回合稳定同时发生。

| 章节 | 普通总生命 | 精英总生命 | Boss 总生命 | 普通峰值伤害 | Boss 峰值伤害 |
| --- | ---: | ---: | ---: | ---: | ---: |
| 第一章 | 55-86 | 82-108 | 100-116 | 12-23 | 18-24 |
| 第二章 | 88-112 | 104-122 | 114-136 | 17-29 | 22-29 |
| 第三章 | 96-122 | 100-122 | 126-150 | 20-32 | 24-34 |

当前 Boss 生命为 `forge_bishop=100`、`storm_archon=116`、`nexus_heart=128`。当前地图池有 21 组正式遭遇进入静态章节预算审计，遭遇项为 0 个硬警告。

Boss 每次阶段转场最多立刻施加一种压力类别，类别按护甲、力量、其他状态和状态牌分别统计。`monster_scaling.json.encounter_constraints.boss_phase_enter_max_effect_categories=1` 是硬边界，`NumericalTreeAuditor` 会逐阶段输出类别并在超限时报告 `boss_phase_entry_categories_high`。单次伤害同时跨过多个生命阈值时，`boss_phase_transition_mode=highest_reached_only` 要求只进入最高可达阶段，只结算该阶段的一组转场效果；阶段后的常规行动仍按遭遇预算结算。

### 完整路径预算

`level_tree.json` 的节点预算约束每一条从起点到 Boss 的完整路径，而不是只约束某一层有哪些候选节点：

| 章节 | 普通战 | 精英 | 路线要求 |
| --- | ---: | ---: | --- |
| 第一章 | 3-4 | 0-1 | 同时存在 0 精英安全路线和 1 精英可选路线。 |
| 第二章 | 2-3 | 0-1 | 同时存在 0 精英安全路线和 1 精英可选路线。 |
| 第三章 | 2-3 | 0-1 | 同时存在 0 精英安全路线和 1 精英可选路线。 |

任意完整路径在两个篝火区间内最多连续经过 4 个压力节点；压力节点指普通战、精英或 Boss。宝箱后不得强制进入精英，Boss 前安全窗口必须包含篝火或商店。`BalanceSimulator` 对每个候选执行深度为 3 的路线评分，含当前候选在内最多累计 3 层节点分数；向下一层递归前会投影篝火治疗、宝箱金币/遗物和精英遗物，使后续评分使用更新后的生命、金币和遗物状态。这是模拟器策略，不代表地图 UI 的剩余工作已经完成。

## 7. 局外成长

每章 Boss 提供 2 炉印，完整通关额外 2 炉印，一次三章通关共 8 炉印。每名角色有 5 个顺序节点，成本为 `3+4+5+6+7=25`，约 3.13 次完整通关解锁。

允许的永久效果只有：最大生命、起始金币、初始势能、药水槽和每场开局护甲。禁止永久力量、额外初始能量和每回合持续收益。完整树最多增加的战斗内稳定性仍低于一张基础防御牌。

### 单卡与胜局遥测

每个角色/挑战 case 都输出按卡牌 id 聚合的 `card_telemetry`，覆盖 `offers`、`acquisitions`、`acquisition_sources`、`removals`、`upgrades` 和 `plays`，并保留获取/打出时的胜负对照及 lift 字段。`win_iteration_indices` 按升序记录该 case 的胜局迭代下标，数量必须与 `wins` 相等，可用于跨角色、跨挑战复核同一配对随机环境。升级牌在删牌、升级和打出统计中归一到基础卡牌 id。

### 可执行矩阵字段

`data/config/numerical_tree.json` 的 `campaign_matrix.rows` 固化上述 12 个角色/挑战格。每一行包含角色基础生命、挑战扣血后的有效开局生命、起始牌组效果点、可访问的可审计卡牌数、成长树总成本、普通敌人/Boss/敌伤挑战修正、目标胜率区间、256 次配对种子模拟的 `observed_win_rate`、最终金币、牌组规模和 `risk_flag`。

第二批扩展后的实际可访问卡池为：流亡者 39 张、工匠 49 张、苦修者 48 张，均由 33 张共享可审计牌加角色专属牌组成。矩阵不是运行时配置，而是可复现的平衡基线和审计契约。`tests/test_numerical_balance_matrix.gd` 会验证 3 x 4 笛卡尔积完整、每格字段与实际角色/挑战数据一致，并重新计算单格容差、挑战均值、角色差和难度单调性；计算出的例外必须与三个 `expected_*` 数组完全一致。

### 覆盖清单字段

`audit_inventory` 声明当前数据规模：71 张总卡牌、70 张可审计牌、25 个敌人、22 组遭遇、3 棵成长树、4 个挑战等级，以及关键经济快照。其中当前地图生成配置会抽取 21 组正式遭遇进入静态章节预算报告；全量计数与地图审计计数分别校验。新增卡牌、怪物、遭遇、成长节点或挑战后必须刷新该清单和矩阵，否则 headless 契约测试会失败，避免内容悄然绕过数值验收。

## 10. 调参流程

1. 修改实际 JSON 数值及 `balance_note`。
2. 运行 JSON 与数据完整性测试。
3. 运行静态数值树；除 `audit_inventory.cards.expected_warning_ids` 中经过评审的显式例外外，不允许新增硬警告。
4. 使用至少 128 次配对完整跑团做方向验证。
5. 使用 256 次矩阵确认均值、角色差、难度单调性、最终金币和牌组规模。
6. 检查 `failure_encounters`；单一遭遇占全部失败超过 50% 时优先改遭遇。
7. 人工试玩复核 AI 无法评价的卡牌理解成本、操作反馈和构筑乐趣。
