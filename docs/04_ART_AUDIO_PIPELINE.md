# 美术与音频资源计划

## 1. 美术方向

关键词：暗黑工业、灰烬、熔炉、炼金术、机械教团、冷青色能量。

第一版 UI 使用分层精修 SVG，优先保证信息可读性、暗色工业气质和小屏不溢出。正式商业版仍应沿资源槽位替换为逐张定制插画、正式图标和授权音频。

## 2. MVP 资产清单

- 玩家角色第一版精修立绘 3 张：`player_ember_exile.svg`、`player_arc_tinker.svg`、`player_pyre_ascetic.svg`。
- 普通敌人第一版精修 SVG 14 张。
- 精英敌人第一版精修 SVG 4 张。
- Boss 第一版精修 SVG 3 张。
- 战斗背景 3 张：第一章余烬工坊、第二章电弧塔楼、第三章回路核心。
- 卡牌第一版独立插图 44 张：`assets/art/cards/card_*.svg`。
- 遗物第一版独立图标 18 张：`assets/art/relics/relic_*.svg`。
- 药水第一版独立图标 8 张：`assets/art/potions/potion_*.svg`。
- 问号事件第一版独立插图 16 张：`assets/art/events/event_*.svg`。
- 卡牌模板 5 套：攻击、技能、能力、状态、诅咒。
- 图标 60 个：能量、势能、护甲、状态、意图、遗物。
- 基础特效 10 个：斩击、护盾、灼烧、破甲、抽牌、死亡、遗物触发。

## 2.1 当前资源清单管线

- `data/config/art_assets.json` 是卡牌、遗物、药水、问号事件插图、章节战斗背景和卡牌类型框的表现资源清单。
- `asset_path` 表示当前 Godot 可加载资源，第一版指向分层精修 SVG、每张卡/遗物/药水/事件的独立插图或类型框 fallback。
- `slot_path` 表示未来正式资源的稳定替换槽位，例如 `assets/art/cards/card_*.svg`、`assets/art/relics/relic_*.svg`、`assets/art/potions/potion_*.svg`、`assets/art/events/event_*.svg` 和 `assets/art/backgrounds/battle_bg_*.svg`。
- 每个槽位必须带 `design_note`、`replacement_note` 和 `implementation_note`，用于记录美术意图、替换限制和接入位置。
- `Main.gd` 已从该清单读取手牌、战斗奖励、商店、遗物奖励、药水槽图标、问号事件插图和当前章节战斗背景；旧硬编码路径只作为 fallback。
- 问号事件页必须使用结构化事件面板，包含事件插图、事件名称、事件正文和选择按钮；事件插图必须从 `event_art_slots` 加载。
- 药水槽、商店药水、战斗药水奖励和遗物奖励必须使用结构化图标物品布局，保留图标、名称、类别/价格和描述分区，不能退回纯文字按钮。
- 玩家摘要面板的常驻遗物栏必须从 `relic_icon_slots` 加载图标，并为每个图标提供遗物名称和描述 tooltip；遗物过多时应使用 `+N` 汇总，不得撑破 720p 基准布局。
- 数据完整性测试会校验所有配置卡牌、遗物、药水、事件和章节背景都有资源槽位，且当前 `asset_path` 能被 Godot 识别。

## 2.2 SVG 质量门槛

- 战斗首屏资源不能是单色块或单路径符号。角色立绘、敌人、卡牌插图、卡框、章节背景、事件图、遗物图标和药水图标都必须包含渐变光照、多个图形层、清晰外轮廓和受控高光。
- `tests/test_data_integrity.gd` 会检查关键 SVG 是否可读取、源码量是否足够、是否包含 `linearGradient` 或 `radialGradient`，以及图形元素数量是否达到最低要求。
- fallback 图标允许继续存在，但必须保持“可展示”的完成度；正式版替换时应继续深化每个卡牌、遗物、药水和事件的独立插画。
- 战斗背景必须保持中低对比度，不得把敌人生命、意图徽章、手牌费用和按钮文字淹没。
- 390x640 小屏仍是硬约束：新增美术不能通过固定大尺寸把角色选择、战斗页、地图页、图鉴页或底部操作条撑出视口。

## 3. 完整版资产规模

- 角色立绘 3 到 6 张。
- 敌人 35 到 50 个。
- Boss 6 到 9 个。
- 卡牌插画 150 到 250 张。
- 遗物图标 120 到 200 个。
- 背景 12 到 20 张。
- 事件插图 30 到 50 张。
- 音效 100 个以上。
- BGM 8 到 12 首。

## 4. UI 规则

- 卡牌信息必须优先清晰：费用、名称、类型、描述、升级状态。
- 手牌、战斗奖励、商店卡牌、篝火升级候选和牌组预览卡必须保持固定分区：费用角标、卡名、图像区、类型标签和描述区，避免长描述把按钮撑开。
- 敌人意图必须比敌人装饰更醒目。
- 玩家资源必须固定位置，避免回合中跳动。
- 当前持有遗物必须在玩家摘要区域保持可见，作为构筑状态的一部分；长列表用紧凑图标和溢出计数处理。
- 问号事件不能退回纯文字日志页；事件名称、正文和插图必须作为同一视觉面板呈现。
- 战斗页首屏必须同时容纳玩家摘要、HUD、敌人舞台、日志摘要、手牌和主操作按钮；普通回合不得显示空奖励区。
- 战斗页布局必须以 1280x720 为最低基准，整页高度预算必须包含标题、状态、玩家面板、战场、日志、手牌/奖励和底部按钮。
- 主页面必须保留全局滚动兜底；窄窗口下底部控制条横向滚动，设置页、奖励页、商店页和事件页使用同一套视口安全容器。
- 地图视图必须包在水平滚动容器内；章节层数增加时允许地图本体变宽，但不能把主页面撑出视口。
- 根内容宽度必须预留垂直滚动条空间；角色卡、事件面板、奖励按钮和遗物栏都必须按实际内容区宽度收缩或汇总。
- 角色选择卡和挑战等级按钮不能使用长按钮文本撑开父容器；可见文本必须短，完整说明放入日志或 tooltip，角色卡内部使用受限 Label/TextureRect 结构。
- 数据图鉴必须复用全局滚动和 `HFlowContainer` 操作区；搜索框固定宽度，分类、发现状态、筛选、具体角色筛选和排序按钮使用短文本，条目卡片固定宽度，设计/平衡/实现注释放入 tooltip。
- 图鉴发现状态在卡片内只使用 `已见` / `未见` 短标签，不得使用长说明撑宽卡面；完整解释放在规则文档或 tooltip。
- 图鉴未见条目默认使用短占位标题、空图框和一行解锁说明；`详情` 模式按钮必须保持两行短文本，不得把全显/隐藏说明写进按钮主体。
- 胜利/失败奖励页应隐藏战斗板，让奖励卡、遗物、药水和继续按钮成为主视图，避免被战场区域挤出屏幕。
- 状态图标必须可悬停查看说明。

## 5. 音频规则

音频分为：

- UI 点击。
- 抽牌和弃牌。
- 出牌。
- 攻击命中。
- 获得护甲。
- 状态施加。
- 遗物触发。
- 敌人死亡。
- Boss 阶段转换。

当前第一版已生成 19 个程序化 WAV 音效，覆盖 UI 点击、出牌类型、回合、奖励、药水、命中、格挡、治疗、Boss 阶段、胜利、失败、地图、篝火、商店、保存和错误提示。`AudioManager` 会优先加载 `assets/audio/*.wav`，缺失时才回退到运行时生成音色。

当前第一版也已生成 10 个轻量 BGM 循环，覆盖菜单、地图、普通战斗、Boss、事件、商店、篝火、奖励、胜利和失败。它们的目标是跑通音乐上下文切换、设置保存和正式资源替换槽位，不代表最终商用品质。

正式商业版可以沿现有文件名直接替换为授权或定制音效/BGM；事件 ID、音乐上下文 ID 和文件名必须保持稳定，避免破坏 UI、战斗反馈、音乐切换和自动化测试。

## 5.1 用户设置

- 设置单独保存到 `user://ember_circuit_settings.json`，不混入跑团存档。
- 局外档案单独保存到 `user://ember_circuit_profile.json`，用于成就、通关角色和跨跑团统计，不能被单局读档回滚。
- 第一版设置包含音效开关、主音量、音乐开关、BGM 音量、震屏、受击顿帧、漂浮战斗文字和新手引导开关。
- `AudioManager` 必须读取音频设置；战斗反馈层必须读取表现设置，关闭后不得继续请求对应反馈。
- 设置页必须可从角色选择、战斗、地图、商店、事件、牌组和结算阶段进入，并使用和其他操作页相同的滚动安全布局。
- 新手引导进度保存在同一设置文件的 `tutorial_completed_steps` 中；设置页必须能重置进度，不能影响跑团存档。

## 6. 新手引导规则

- 引导必须是上下文提示，不应遮挡核心交互。
- 当前提示显示在状态栏和 tooltip 中，完整引导目录通过底部“引导”按钮打开。
- 引导步骤覆盖角色选择、玩家战斗回合、战斗奖励、地图选择、问号事件、商店、篝火、牌组查看、设置页和跑团结算。
- 玩家完成当前提示后，该提示在后续刷新中隐藏；设置页可重新启用和重置所有提示。
