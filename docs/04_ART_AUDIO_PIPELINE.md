# 美术与音频资源计划

## 1. 美术方向

关键词：暗黑工业、灰烬、熔炉、炼金术、机械教团、冷青色能量。

第一版 UI 使用分层精修 SVG，优先保证信息可读性、暗色工业气质和小屏不溢出。正式商业版仍应沿资源槽位替换为逐张定制插画、正式图标和授权音频。

## 2.1 当前资源清单管线

- `data/config/art_assets.json` 是卡牌、药水、章节战斗背景、角色/敌人舞台切图、HUD 纹理、遗物、问号事件插图和卡牌类型框的表现资源清单，也是 PC 位图生产契约的唯一配置源。
- `asset_path` 表示当前 Godot 可加载资源，可以是生产位图，也可以是仍待替换的首版 SVG fallback。
- `slot_path` 表示未来正式资源的稳定替换槽位，例如 `assets/art/cards/card_*.svg`、`assets/art/relics/relic_*.svg`、`assets/art/potions/potion_*.svg`、`assets/art/events/event_*.svg` 和 `assets/art/backgrounds/battle_bg_*.svg`。
- 每个槽位必须带 `design_note`、`replacement_note` 和 `implementation_note`，用于记录美术意图、替换限制和接入位置。
- `Main.gd` 已从该清单读取手牌、战斗奖励、商店、遗物奖励、药水槽图标、问号事件插图和当前章节战斗背景；旧硬编码路径只作为 fallback。
- `player_stage_slots`、`enemy_stage_slots` 和 `hud_texture_slots` 当前用于生产审计，并与 `Main.gd` 的现有同路径常量保持一致；运行时改为完全数据驱动属于后续接线任务。
- 问号事件页必须使用结构化事件面板，包含事件插图、事件名称、事件正文和选择按钮；事件插图必须从 `event_art_slots` 加载。
- 药水槽、商店药水、战斗药水奖励和遗物奖励必须使用结构化图标物品布局，保留图标、名称、类别/价格和描述分区，不能退回纯文字按钮。
- 玩家摘要面板的常驻遗物栏必须从 `relic_icon_slots` 加载图标，并为每个图标提供遗物名称和描述 tooltip；遗物过多时应使用 `+N` 汇总，不得撑破 720p 基准布局。
- 数据完整性测试会校验所有配置卡牌、遗物、药水、事件和章节背景都有资源槽位，且当前 `asset_path` 能被 Godot 识别。

## 2.2 PC 位图资产契约

`asset_contract` 记录 1280x720 PC 布局基准、section 默认生产级别和逐类位图规则。布局基准用于裁切与安全区复核，不要求所有源图都输出为 1280x720。

| 类别 | 源图尺寸与比例 | 通道/透明度 | 构图门槛 |
| --- | --- | --- | --- |
| 战斗背景 | `1920x640`，比例 `3:1`，exact | RGB，禁止 alpha | 中部 70% x 72% 是角色、意图与生命信息安全区；环境焦点占比 15%-65%，人工复核 |
| 角色舞台切图 | 最小 `640x720`，比例 0.55-1.15 | RGBA，必须存在真实透明像素 | alpha 包围盒留 1% 边距，占画布包围盒面积 20%-90% |
| 敌人舞台切图 | 最小 `640x640`，比例 0.65-1.50 | RGBA，必须存在真实透明像素 | alpha 包围盒留 1% 边距，占画布包围盒面积 18%-90% |
| 卡牌插画 | `784x1168`，比例 `0.671233`，exact | RGB，禁止 alpha | 中部 84% x 84% 为动作安全区；禁止文字、费用、卡框、Logo 和水印，人工复核 |
| 事件插画 | `1536x1024`，比例 `3:2`，exact | RGB，禁止 alpha | 中部 80% x 84% 为主体安全区；适配 PC 左侧画框 cover 裁切，禁止文字、卡框、Logo 和水印 |
| 药水图标 | 最小 `1024x1024`，比例 1:1 | RGBA，必须存在真实透明像素 | alpha 包围盒留 2% 边距，占画布包围盒面积 25%-82% |
| HUD 纹理 | 每个 `hud_texture_slots[].contract_id` 定义 exact 尺寸/比例 | 当前槽位为 RGB，禁止 alpha | 每个控件分别定义文字/数值安全区与主体占比，人工复核 |

生产级别规则：

- `production_required`：发布阻断级。资源缺失、无法解码、扩展名、尺寸、比例、RGB/RGBA 或 alpha 不合约均记为 hard error。
- `production_preferred`：已经进入 PC 生产管线的位图同样严格检查。允许 fallback 的 section 遇到可加载 SVG 时改记为 `legacy_fallback`，不会制造整批硬失败。
- `legacy_fallback`：首版 SVG 或旧资源可以继续展示，但审计器会给出替换 advisory。fallback 文件本身缺失仍是 hard error。

`ArtAssetAuditor` 直接按 PNG/WebP/JPEG 源文件字节解码，不使用导入后可能改变格式的纹理。报告包含实际 `width`、`height`、`aspect_ratio`、`color_mode`、`has_alpha_channel`、`uses_transparency`、`alpha_mode` 和透明主体 `subject_bounds`。尺寸、比例、通道和 alpha 是机器硬校验；RGB 图的语义安全区/主体占比是 manual advisory，RGBA 切图则用 alpha 可见包围盒自动给出构图 advisory。

运行测试与完整审计：

```bash
/Applications/Godot.app/Contents/MacOS/Godot --headless --path . --script res://tests/test_art_asset_auditor.gd
/Applications/Godot.app/Contents/MacOS/Godot --headless --path . --script res://tools/run_art_asset_audit.gd
```

审计进程本身不会因清单内存在生产缺口退出非零；CI 是否阻断应读取 `summary.hard_failures`。测试负责保证 schema、像素读取、严格/降级语义和真实清单覆盖不回退。

## 2.3 SVG 质量门槛

- 战斗首屏资源不能是单色块或单路径符号。角色立绘、敌人、卡牌插图、卡框、章节背景、事件图、遗物图标和药水图标都必须包含渐变光照、多个图形层、清晰外轮廓和受控高光。
- `tests/test_data_integrity.gd` 会检查关键 SVG 是否可读取、源码量是否足够、是否包含 `linearGradient` 或 `radialGradient`，以及图形元素数量是否达到最低要求。
- fallback 图标允许继续存在，但必须保持“可展示”的完成度；正式版替换时应继续深化每个卡牌、遗物、药水和事件的独立插画。
- 战斗背景必须保持中低对比度，不得把敌人生命、意图徽章、手牌费用和按钮文字淹没。
- 三章当前正式背景均由 `gpt-image-2` 生成，再确定性裁切为 `1920x640 RGB`；源图不直接进入运行时，避免不同上游尺寸破坏舞台构图。
- 中文 UI 使用随包分发的 `assets/fonts/NotoSansSC-Variable.ttf`，授权文本为 `assets/fonts/OFL.txt`；禁止依赖玩家系统字体，以免出现缺字方框或不同平台排版漂移。
- 390x640 小屏仍是硬约束：新增美术不能通过固定大尺寸把角色选择、战斗页、地图页、图鉴页或底部操作条撑出视口。

## 4. UI 规则

- 卡牌信息必须优先清晰：费用、名称、类型、描述、升级状态。
- 手牌、战斗奖励、商店卡牌、篝火升级候选和牌组预览卡必须保持固定分区：费用角标、卡名、图像区、类型标签和描述区，避免长描述把按钮撑开。
- 敌人意图必须比敌人装饰更醒目。
- 玩家资源必须固定位置，避免回合中跳动。
- 当前持有遗物必须在玩家摘要区域保持可见，作为构筑状态的一部分；长列表用紧凑图标和溢出计数处理。
- 问号事件不能退回纯文字日志页；事件名称、正文和插图必须作为同一视觉面板呈现。
- PC 问号事件使用单一固定舞台：左侧为 `3:2` 插画、标题和叙事，右侧为纵向决策列；至少四个选项必须在 `1280x720` 内一次显示完整，禁止依靠页面滚动条访问选项。
- PC 事件选项使用独立黄铜/炭黑状态皮肤和编号徽章，不复用通用紫色事件按钮；悬停、按下、键盘焦点和条件不足状态必须可区分。
- 战斗页首屏必须同时容纳玩家摘要、HUD、敌人舞台、日志摘要、手牌和主操作按钮；普通回合不得显示空奖励区。
- 战斗页布局必须以 1280x720 为最低基准，整页高度预算必须包含标题、状态、玩家面板、战场、日志、手牌/奖励和底部按钮。
- PC 普通战斗的手牌尺寸应保持稳定，额外纵向空间由战斗舞台吸收；1280x720 与 1600x900 下手牌装置区和底部控制条之间不得留下无用途空洞。舞台背景必须使用 cover 裁切且忽略纹理原始最小尺寸，不能随舞台增高反向撑宽页面。
- 主页面必须保留全局滚动兜底；窄窗口下底部控制条横向滚动，设置页、奖励页、商店页和事件页使用同一套视口安全容器。
- 地图视图必须包在水平滚动容器内；章节层数增加时允许地图本体变宽，但不能把主页面撑出视口。
- 根内容宽度必须预留垂直滚动条空间；角色卡、事件面板、奖励按钮和遗物栏都必须按实际内容区宽度收缩或汇总。
- 角色选择卡和挑战等级按钮不能使用长按钮文本撑开父容器；可见文本必须短，完整说明放入日志或 tooltip，角色卡内部使用受限 Label/TextureRect 结构。
- 数据图鉴必须复用全局滚动和 `HFlowContainer` 操作区；搜索框固定宽度，分类、发现状态、筛选、具体角色筛选和排序按钮使用短文本，条目卡片固定宽度，设计/平衡/实现注释放入 tooltip。
- 图鉴发现状态在卡片内只使用 `已见` / `未见` 短标签，不得使用长说明撑宽卡面；完整解释放在规则文档或 tooltip。
- 图鉴未见条目默认使用短占位标题、空图框和一行解锁说明；`详情` 模式按钮必须保持两行短文本，不得把全显/隐藏说明写进按钮主体。
- 胜利/失败奖励页应隐藏战斗板，让奖励卡、遗物、药水和继续按钮成为主视图，避免被战场区域挤出屏幕。
- 最终结算页必须使用独立结局布局，不得把长总结文本和动作按钮直接混在奖励流里；PC 端至少检查 1600x900 和 1280x720 截图，确保结局动作按钮完整可见。
- 状态图标必须可悬停查看说明。

### 4.1 全页面视觉与性能门

- 标准 PC 视觉合同覆盖欢迎、角色、战斗、奖励、地图、事件、商店、篝火、结算、设置和图鉴 11 页，固定为 1280x720，并按页面语义区域比较，不使用宽松全屏阈值。
- 图库每次截图前重置全局随机种子，避免初始手牌洗牌导致金标漂移；战斗页另保留 1600x900 人工构图检查。
- reduced motion、闪光强度和粒子密度属于全局表现策略；降低动态效果必须停止持续位移、缩放和粒子，同时保留不依赖运动的可见确认。
- 性能采样必须实例化真实 `Main.tscn`，经过 20 轮 map->event->shop->campfire->reward 切换后重开真实战斗，并检查 600 帧 p95、1% low、输入反馈、节点增量、循环 Tween 与粒子预算。

### 4.2 018D 跑团页面视觉合同

- Reward 固定为结算收据、战利品选择和命令栏三段；卡牌数量较少时卡面纵向填满可用区域，不能在卡牌行下留下无用途空洞。继续路线保持右下固定，并严格反映 Main 的奖励门。
- Event 左侧使用事件生产插画、标题和正文，右侧显示完整选择列；Campfire 左侧使用房间生产插画，右侧显示生命状态、休息/锻造和离开动作。
- Shop 三个货架必须为每件卡牌、遗物和药水显示现有生产插画/图标、价格和不可购买原因；不同库存数量由货架按钮平均吸收高度，不以空面板填充。
- Map 保持完整路线图、风险/收益与后继详情，不引入第二个活动 MapView。
- 五页同时检查 1280x720 与 1600x900 构图；区域金标只更新 `03_reward` 至 `07_campfire`，欢迎、角色、战斗、结算、设置和图鉴六页必须像素稳定。阈值保持 mean RGB <=1%、changed pixels <=2%。

## 5. 音频规则

音频分为：

- UI 点击。
- 抽牌和弃牌。
- 出牌。
- 攻击命中。
- 获得护甲。
- 状态施加。
- 遗物触发。
- 敌人死亡。
- Boss 阶段转换。

当前第一版已生成 19 个程序化 WAV 音效，覆盖 UI 点击、出牌类型、回合、奖励、药水、命中、格挡、治疗、Boss 阶段、胜利、失败、地图、篝火、商店、保存和错误提示。`AudioManager` 会优先加载 `assets/audio/*.wav`，缺失时才回退到运行时生成音色。

当前第一版也已生成 10 个轻量 BGM 循环，覆盖菜单、地图、普通战斗、Boss、事件、商店、篝火、奖励、胜利和失败。它们的目标是跑通音乐上下文切换、设置保存和正式资源替换槽位，不代表最终商用品质。

正式商业版可以沿现有文件名直接替换为授权或定制音效/BGM；事件 ID、音乐上下文 ID 和文件名必须保持稳定，避免破坏 UI、战斗反馈、音乐切换和自动化测试。

## 5.1 用户设置

- 设置单独保存到 `user://ember_circuit_settings.json`，不混入跑团存档。
- 局外档案单独保存到 `user://ember_circuit_profile.json`，用于成就、通关角色和跨跑团统计，不能被单局读档回滚。
- settings schema v2 包含音效开关、主音量、音乐开关、BGM 音量、震屏、受击顿帧、漂浮战斗文字、新手引导、降低动态效果、闪光强度和粒子密度；强度字段固定为 0.25 步长并 clamp 到 0..1。
- `AudioManager` 必须读取音频设置；战斗反馈层必须读取表现设置，关闭后不得继续请求对应反馈。
- 设置页必须可从角色选择、战斗、地图、商店、事件、牌组和结算阶段进入，并使用和其他操作页相同的滚动安全布局。
- 新手引导进度保存在同一设置文件的 `tutorial_completed_steps` 中；设置页必须能重置进度，不能影响跑团存档。

## 6. 新手引导规则

- 引导必须是上下文提示，不应遮挡核心交互。
- 当前提示显示在状态栏和 tooltip 中，完整引导目录通过底部“引导”按钮打开。
- 引导步骤覆盖角色选择、玩家战斗回合、战斗奖励、地图选择、问号事件、商店、篝火、牌组查看、设置页和跑团结算。
- 玩家完成当前提示后，该提示在后续刷新中隐藏；设置页可重新启用和重置所有提示。
